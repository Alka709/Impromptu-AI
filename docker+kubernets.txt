# Impromptu-AI: Docker and Kubernetes Architectural Approach

This document outlines the exact architectural decisions, containerization strategies, and Kubernetes orchestration approaches taken for the Impromptu-AI platform. The focus is on explaining *what* we did and, more importantly, *why* we took these specific approaches over alternatives, particularly regarding AI model handling and Windows/Linux cross-compatibility.

---

## 1. Overall System Architecture

The Impromptu-AI platform is built on a microservices architecture to decouple the fast, I/O-bound web traffic (Node.js) from the heavy, CPU/GPU-bound speech analysis tasks (Python/AI).

**The stack consists of 5 core containers:**
1. **Frontend**: React (Vite) served via NGINX.
2. **Backend**: Node.js/Express serving as the main API and BFF (Backend-for-Frontend).
3. **AI Service**: Python/FastAPI exposing synchronous AI endpoints (like Gemini topic generation).
4. **AI Worker**: Python/RQ running as a background job consumer to process heavy audio evaluations.
5. **Redis**: An in-memory message broker coordinating jobs between the Backend and the AI Worker.

*(Note: PostgreSQL/Supabase and AWS S3 are used as external managed services and are purposely excluded from the containerization stack to ensure statelessness).*

---

## 2. Dockerization Strategy & Decisions

### A. Frontend (React/Vite)
* **Approach**: Multi-stage build using `node:20-alpine` to compile the static assets, and `nginxinc/nginx-unprivileged:alpine` to serve them.
* **Why this approach?** 
  * **Why NGINX?** Vite is a development tool. In production, React compiles to raw HTML/JS/CSS. NGINX is the industry standard for serving static files blazing fast.
  * **Why `nginx-unprivileged`?** Standard NGINX runs as `root`, which poses severe security risks and causes permission denied errors on Windows Docker Desktop. Using the unprivileged image fixes cross-platform permission crashes instantly.
  * **The SPA Routing Fix**: We explicitly inject a custom `nginx.conf` containing `try_files $uri $uri/ /index.html;`. Without this, if a user manually refreshes a page like `/sessions/123`, NGINX attempts to find a literal `123.html` file and throws a 404 error. This custom config forces NGINX to route all unknown paths back to React's internal router.

### B. Backend (Node.js)
* **Approach**: Single-stage build on `node:20-alpine`. Mapped explicitly to **Port 4000** in Docker Compose.
* **Why this approach?**
  * **Port Unification**: Initially, Docker ran the backend on port 3000. However, the local `.env` files, Frontend API variables (`VITE_API_BASE`), and Google OAuth Callbacks all hardcoded `http://localhost:4000`. By unifying Docker to run on 4000, developers can seamlessly switch between native Windows execution and Docker Compose without having to rewrite `.env` files or break Google Login redirects.

### C. AI Service & AI Worker (Python)
* **Approach**: Both the FastAPI service and the RQ Worker share the *exact same* Docker image (`python:3.11-slim`), but `docker-compose.yml` and Kubernetes overwrite the `command:` to dictate how the container behaves.
* **Why this approach?** 
  * Sharing an image saves massive amounts of disk space and build time since the Python dependencies (PyTorch, Whisper, FFmpeg) only need to be downloaded and compiled once.

#### Critical AI Decision 1: Pre-Baking Models at Build Time
* **The Problem**: Faster-Whisper and Silero VAD require downloading ~1.5GB of model weights from HuggingFace. If we let the container download these at runtime, every time you spin up a new container it takes 3-5 minutes to become usable.
* **The Solution**: We added `RUN python -c "from faster_whisper import WhisperModel..."` directly into the `Dockerfile`.
* **Why not use a shared volume instead?** In local Docker, a shared volume (`hf_cache`) is fine. But in Production Kubernetes, if traffic spikes and K8s autoscales from 2 to 10 pods, you want those 8 new pods to process requests instantly (0 seconds). If they rely on a shared volume, they might bottleneck on network I/O. If they download at runtime, you suffer a 5-minute lag. "Baking" the models into the Docker image guarantees instant, bulletproof scaling.

#### Critical AI Decision 2: The Forking Deadlock Fix (`rq.SimpleWorker`)
* **The Problem**: The default RQ worker uses `os.fork()` to create a child process for every job. While this works well for simple tasks, Python's `os.fork()` is notoriously unstable when interacting with heavy C++ multi-threading libraries like PyTorch and CTranslate2 (used by Whisper). Forking *after* thread pools are initialized causes silent deadlocks—the worker just hangs forever.
* **The Solution**: We explicitly appended `--worker-class rq.SimpleWorker` to the Docker and K8s commands.
* **Why this approach?** `SimpleWorker` executes the evaluation job in the main thread, completely avoiding the fork. Since our architecture relies on K8s to scale out horizontally (1 pod = 1 worker), we do not need RQ to spawn child processes. The `SimpleWorker` ensures absolute stability for PyTorch/Whisper in Linux containers.

## 3. Kubernetes (K8s) Orchestration Approach

Our Kubernetes manifests (`/k8s`) are designed for a scalable production environment.

### A. Environment Variable Injection
* **Approach**: We use a K8s `ConfigMap` for non-sensitive data (Ports, internal K8s URLs) and a `Secret` for sensitive keys (Gemini API, AWS, Supabase).
* **Why?** This abstracts configuration away from the code. The Python and Node.js containers don't know they are in Kubernetes; they just read standard environment variables.

### B. Internal Service Discovery
* **Approach**: Instead of IP addresses or `localhost`, microservices talk to each other using Kubernetes internal DNS (e.g., `http://backend-service:4000` or `redis://redis-service:6379`).
* **Why?** Kubernetes Pods are ephemeral; their IPs change constantly as they crash or scale. The `Service` acts as a permanent load balancer, ensuring that the AI Worker can always find the Backend, no matter how many Pods are currently alive.

### C. Resource Limits & Requests
* **Approach**: The AI Worker is configured with strict CPU/Memory requests (`250m` / `512Mi`) and limits (`1000m` / `1Gi`).
* **Why?** PyTorch and Whisper will aggressively consume all available CPU and RAM if left unchecked, starving the Node.js backend on the same cluster. Strict limits prevent the "Noisy Neighbor" K8s problem, ensuring cluster stability.

### D. Why No Database in Kubernetes?
* **Approach**: We did not deploy a PostgreSQL or S3 pod in our K8s cluster.
* **Why?** Managing stateful databases in Kubernetes (handling Persistent Volumes, automated backups, replication) is notoriously difficult and risky. By relying on Supabase (Managed Postgres) and AWS S3, our K8s cluster remains 100% **stateless**. If the cluster catches fire, we simply redeploy the YAML files to a new cluster with zero data loss.

---

## Conclusion
This architecture provides the perfect balance between **Developer Experience** (unifying ports for easy local dev) and **Production Readiness** (pre-baked models, SimpleWorker dead-lock prevention, and stateless K8s scaling). We actively avoided "cutting corners" (like runtime AI downloads) to ensure the system is truly enterprise-ready.
