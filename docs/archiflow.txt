====================================================================
DETAILED ARCHITECTURE AND SYSTEM MECHANICS ANALYSIS: MYPROMPTU
====================================================================

1. HOW EVERYTHING IS ACTUALLY WORKING (System Mechanics)
--------------------------------------------------------------------
The Mypromptu platform operates on a polyglot microservices architecture, splitting responsibilities between a primary Web API and a background AI/Audio processing pipeline.

Step-by-Step Flow:
- **Authentication & Web Routing (Node.js/Express)**: When a user interacts with the React frontend, all standard web traffic hits the Node.js server. Node.js acts as the gateway. It handles JWT authentication, validates requests, and reads/writes to the PostgreSQL database using Drizzle ORM.
- **Topic Generation (FastAPI - Synchronous)**: When the user requests a speech topic, Node.js forwards this request to the Python FastAPI microservice. Python securely queries the Google Gemini API, gets a topic, and returns it to Node.js, which then passes it to the frontend.
- **Audio Hand-off (AWS S3)**: When a user finishes recording, the frontend requests a "Pre-Signed PUT URL" from the Node.js backend and securely uploads the massive raw audio file *directly* to AWS S3, bypassing the Node.js memory entirely. The frontend then confirms the upload with Node.js, which generates a temporary "Pre-Signed GET URL" valid for 15 minutes.
- **Asynchronous Handoff (Redis & RQ)**: Node.js sends the Pre-Signed URL, the user's Session ID, and their past evaluation history to FastAPI via an HTTP trigger (`/api/evaluate/enqueue`). FastAPI doesn't process the audio immediately. Instead, it places a "Job" into a Redis Queue. Node.js instantly returns a success response to the user so they aren't kept waiting.
- **Background Worker Processing (Python RQ)**: A separate Python worker process constantly listens to the Redis queue. When it sees the job, it:
    1. Downloads the audio from S3 using the Pre-Signed URL.
    2. Calls Groq's blazing-fast Whisper API for transcription (eliminating local OOM risks).
    3. Runs Librosa/Parselmouth for mathematical audio analysis (pitch, filler words, energy).
    4. Evaluates the speech in parallel using a Multi-Agent Evaluation Engine (Content Agent, Delivery Agent, Overall Coach) backed by Gemini, integrating the user's history directly from the job payload.
- **Webhook Delivery (Tenacity)**: Once the Python worker finishes the evaluation, it sends the final JSON report via an HTTP POST (Webhook) back to the Node.js server (`/api/webhooks/evaluation-result`), wrapped in an exponential backoff retry loop using `tenacity` to ensure robust delivery. Node.js saves it in the PostgreSQL database, ready for the user to view.


2. WHY DID WE EXPLICITLY CHOOSE THIS ARCHITECTURE?
--------------------------------------------------------------------
The architecture was chosen to adhere to the principle of "using the right tool for the job."

- **Node.js for the Web Layer**: Node.js uses an event-driven, non-blocking I/O model. It is exceptionally fast at handling thousands of concurrent lightweight web requests (routing, database queries, JWT validation). 
- **Python for AI and Audio**: Python is the undisputed king of Machine Learning and Audio processing. Libraries like `librosa` (audio analysis), `numpy`, and `openai-whisper` have deep integrations in Python. Trying to do complex audio math in JavaScript is highly inefficient and lacks mature libraries.
- **Redis Queue (Asynchrony)**: Audio transcription and LLM evaluation take 15 to 45 seconds. If we ran this synchronously, the HTTP request from the frontend would time out (browsers usually timeout after 30 seconds), and the Node.js event loop would be blocked. The queue decouples the fast web layer from the slow processing layer.
- **AWS S3 as Intermediary**: Sending large binary blobs directly between microservices consumes massive amounts of RAM and network bandwidth, risking Out-Of-Memory (OOM) crashes. S3 acts as a stateless, cheap, and infinitely scalable intermediary buffer.


3. WHAT OTHER OPTIONS WERE THERE? (And why we rejected them)
--------------------------------------------------------------------
Option A: The Monolith (All in Node.js)
- *How it works*: Everything, including audio processing, runs in the Express server.
- *Why we rejected it*: Node.js is single-threaded. CPU-bound tasks like processing audio matrices with Librosa or running Whisper would completely freeze the server, causing all other users to experience hanging requests. Furthermore, native JS audio libraries are severely lacking compared to Python's ecosystem.

Option B: The Monolith (All in Python/Django/FastAPI)
- *How it works*: Node.js is removed entirely. FastAPI handles the frontend, the database, auth, and the AI.
- *Why we rejected it*: While simpler to deploy, Python web frameworks are traditionally slower for raw web I/O compared to Node.js. Additionally, managing heavy background tasks in the same process as the web server in Python still requires a task queue (like Celery/RQ), meaning we don't actually escape the complexity of a queue system.

Option C: Synchronous Microservices (No Redis/RQ)
- *How it works*: Node.js calls Python, and waits for 40 seconds for Python to finish processing, then returns the result to the user.
- *Why we rejected it*: Terrible User Experience (UX). Users staring at a spinning wheel for a minute is bad practice. Browsers or proxies (like Nginx) will aggressively kill connections that take that long.

Option D: Storing Audio in the PostgreSQL Database (Bytea)
- *How it works*: Saving the raw `.webm` audio files as binary data in the SQL database instead of AWS S3.
- *Why we rejected it*: Storing massive files in SQL rapidly bloats the database, drastically degrades query performance, and makes backups incredibly slow and expensive. S3 is designed specifically for cheap, large object storage.


4. BENEFITS AND LOSSES (Trade-offs)
--------------------------------------------------------------------
### The Benefits (Why we win)
- **Massive Scalability**: The system is horizontally scalable. If thousands of users upload audio simultaneously, Node.js easily handles the web traffic, S3 easily handles the storage, and jobs just queue up in Redis. If the queue gets too long, you can spin up 10 more Python RQ Workers on different servers to burn through the queue, without touching the Node.js backend.
- **Uncompromised UX**: The user interface never freezes. Users get instant feedback that their upload was successful while the heavy lifting happens in the background.
- **Best-in-Class Libraries**: We don't have to compromise on features. We get the speed of Express and the analytical power of Python's scientific ecosystem.
- **Stateless Microservices**: Because of S3 Pre-Signed URLs and Webhooks, the Python worker doesn't need to share a hard drive or database connection with Node.js. It only needs an internet connection.

### The Losses (What we sacrificed)
- **High Operational Complexity**: Running this locally or deploying it to production requires orchestrating multiple moving parts (Node server, React frontend, FastAPI server, Python Worker, PostgreSQL DB, Redis Server, AWS S3). 
- **Harder Debugging (Distributed Tracing)**: If a request fails, you have to check multiple logs. Did it fail in Node? Did the S3 upload fail? Did the RQ Worker crash? Did the Webhook fail to deliver? Debugging requires tracing IDs across service boundaries.
- **Eventual Consistency Issues**: Because the system is asynchronous, the frontend has to either poll the backend (`GET /api/sessions/:id/evaluation`) or rely on WebSockets to know when the background job is done. This adds complexity to the frontend state management.
